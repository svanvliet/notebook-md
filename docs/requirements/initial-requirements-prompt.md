I’d like to create an app called Notebook.md that enables a user to create notebooks comprised of Markdown files that they can create, edit, and share in an easy-to-use canvas. The primary goal of this app is to be simple to use, simple to deploy, and leverage existing systems for storage and sharing. I want the app to be deployable to a central website where users can create an account using credentials from Microsoft (either personal/MSA or their enterprise/M365 accounts), GitHub, Google, Apple, or with their email address. That means we’ll need to have a central user account system but would only use that for storing metadata and account linkages. We’ll use best practices for this approach so that we minimize security risks and only store what’s absolutely necessary.
I’m envisioning an interface with a title/tool bar along the top, a workspace pane on the left, and a main canvas, the document pane, on the right taking up the majority of the screen. The workspace pane will have a tree view of workspaces you can open, and under each workspace, you’ll see a hierarchy of Notebooks, which would be folders containing Markdown files and any number/depth of nested sub-folders with files. A workspace would be a connection to a location one of the various sources, tied to credentials from the aforementioned services. For example, if you authenticated with Microsoft, you could add a OneDrive folder as a Notebook. Same with Google for Google Drive location, Apple would be an iCloud folder, etc. For GitHub, it would be a repo that you have access to; you could select an entire repo and the app would only show .md files, or you could specify a specific folder of the GitHub repo as the Notebook root.
Once the user adds a workspace, they can add a new, open an existing, .md file from that location and it will open in the document pane. The view would be a rendered Markdown view that would be a live editor, in a WYSIYG fashion. Users can type directly in that view, editing existing text and Markdown elements. All primary Markdown items should be supported. As the user is editing the document, the changes should be captured by the client, and the user has the option to save. Alternatively, users should be able to enable auto-save for a workspace, so it automatically saves their changes as they edit. The app should be smart about how often it saves changes so as not to spam the workspace source system (you can decide which approach to take by source system). For version-controlled sources, like GitHub, please suggest some approaches for how saves should be handled (e.g., should the app treat the workspace in the app like a git checkout of those files, and then stage changes locally before pushing a commit to origin, or do some other approach towards saving in a branch, etc.) Be creative here. Each workspace in the workspace pane should have a little icon indicating it’s source type (e.g. Octocat for Github, the OneDrive cloud icon for M365, and Apple icon for iCloud).
Users should be able to have multiple documents open in the notebook, so we should support a tabbed view in the document pane. The text on that tab should have the name of the file that’s opened. When a document is open, the toolbar should be updated to reflect some of the popular Markdown features for headings, lists, tables, etc. The document editor should also support slash (/) commands so that you can change the format of text or insert a Markdown control. 
The app should have an account dropdown on the top-right of the tool bar that gives a drop down with options to manage their account details, linkages to source systems, etc. via a modal dialog. Users can add/edit/remove account linkages from here. It should also have a Settings item that opens a modal dialog for changing their general settings and preferences. This would include things like display mode (light, dark, system) which would affect how the app renders, font preferences for display in the editor, document margins (regular, wide, narrow, etc.) which affects how the documents are displayed, etc.
The app should start with a welcome screen that takes users through the sign-in, sign-up flow. You can mirror common flows for this from other apps for this feature. Use a notebook icon for the logo for now, but I will replace it with a new logo I’ll build for it. A smaller version of that icon can be placed in the top bar of the app when signed in next to the next Notebook.md, which is the header of the app.
The bottom of the app should have a very thin status bar that spans the wide of the page. We’ll use that area to show live document stats, like work count and last saved time stamp. We can also display ephemeral messages there as needed.
Finally, I want this app to run on the web at notebookmd.io, a domain I’ve purchased from Godaddy. I want to be able to deploy easily locally for development, and then to either Azure or AWS for the production deployment. Ideally, we’d use containers for hosting and deployment of the app, so things are modular. Suggest if this should be a single container or deployed with multiple containers on something like Kubernetes. Tell me how I can build it agnostic of cloud provider, but if there are benefits to using one provider’s specific tools, explain why. I also want to have versions of the app that run natively on Mac and Windows. Please give me options for doing so. It would be great to use the same codebase for the Web, Mac, and Windows versions, but I’m also okay considering a specific codebase for the native apps and a single codebase for the web app. Not for now, but I will want mobile apps for iOS and Android, so consider that when weighing your recommendations for the native Mac and Windows apps should that decision impact future mobile app choices.
