# Notebook.md — Security Review of Requirements Document

**Reviewed:** `requirements/requirements.md` v1.3 (2026-02-17)  
**Review Date:** 2025-07-18  
**Scope:** Security and deployment risks in the product requirements

---

## Critical Severity

### 1. Automatic Account Merging Enables Account Takeover

**File:** requirements.md §2.3, line 72  
**Severity:** Critical

**Problem:** The requirement states that if a user signs in with Provider A (e.g., Google, `user@gmail.com`) and later signs in with Provider B (e.g., Email, `user@gmail.com`) using the same verified email, accounts are automatically merged. This is a well-known account takeover vector.

An attacker who knows a victim's email address can:
1. Create a Notebook.md account via email+password sign-up using the victim's email address *before* the victim signs up
2. The victim later signs in via Google OAuth with the same email
3. The system auto-merges, giving the attacker's pre-existing email+password credentials access to the victim's OAuth-linked account (and all their OAuth tokens/notebooks)

This attack works because the email+password path lets anyone claim an email address. The "verified email" qualifier is insufficient — a magic link or email verification confirms the *attacker* controls the email *at that time*, but doesn't establish identity equivalence with a future OAuth sign-in.

**Recommendation:** 
- Never auto-merge when the incoming provider is OAuth and an existing email+password account exists (or vice versa) unless the user explicitly confirms the merge while authenticated on *both* providers in the same session.
- Auto-merge should only occur between two OAuth providers where the email is verified by the provider (Google, Microsoft, Apple — where the `email_verified` claim is `true`).
- For email+password ↔ OAuth merges, require the user to log in with one provider and then explicitly link the other from Account Settings.

### 2. OAuth Tokens Sent to the Browser — Full Token Exposure

**File:** requirements.md §8.2, lines 537–539  
**Severity:** Critical

**Problem:** The architecture states: *"The client communicates directly with source system APIs for file operations using tokens obtained via the backend."* This means OAuth access tokens (and potentially refresh tokens) for OneDrive, Google Drive, and GitHub are transmitted to and held in the browser/client.

This creates a massive blast radius:
- **XSS = full account compromise:** Any XSS vulnerability in the app (or any dependency — Tiptap, ProseMirror, etc.) gives the attacker the user's OAuth tokens for Microsoft, Google, and GitHub. These tokens provide read/write access to the user's OneDrive files, Google Drive files, and GitHub repos — far beyond the Notebook.md application itself.
- **Token exfiltration via browser extensions:** Malicious or compromised browser extensions can read tokens from JavaScript memory, localStorage, or IndexedDB.
- **Token logging:** Client-side error tracking (Sentry) could inadvertently capture tokens in breadcrumbs, network requests, or error payloads.
- The document claims "no document content stored centrally" as a security benefit, but this architecture trades that for something worse: the client holds the keys to *all* the user's cloud storage.

**Recommendation:**
- Route all source system API calls through the backend proxy. The backend holds the OAuth tokens; the client never sees them. This is the standard architecture for OAuth-based applications.
- If direct client access is required for latency reasons, use short-lived, narrowly-scoped proxy tokens or signed URLs generated by the backend (e.g., OneDrive's pre-authenticated download URLs, Google Drive's short-lived access URLs). Never expose the full OAuth token to the client.
- The CORS note in §8.2 already acknowledges the backend may need to proxy. Make this the default, not the exception.

### 3. Dev Mode Exposes Secrets and Attack Surface in Production

**File:** requirements.md §8.5, lines 604–610  
**Severity:** Critical

**Problem:** The `DEV_MODE` flag (§8.5 line 604) and account-level `is_dev_mode` flag (§8.5 line 610) enable dangerous behaviors:

1. **Magic links and 2FA codes logged to API console output** (§2.2, line 64): If `DEV_MODE` is accidentally enabled in production (misconfigured env var), all magic links and 2FA codes are written to logs. Anyone with log access (cloud console, SIEM, log aggregation) can hijack any account.

2. **Account-level dev mode in production** (line 610): A database flag (`is_dev_mode`) enables "debug panel, verbose logging" for specific accounts *in production*. This means:
   - Detailed error messages and stack traces are returned from the API for dev-mode accounts, leaking internal implementation details
   - The debug panel could expose internal state, API responses, token metadata
   - An admin who is compromised (or goes rogue) can flag any account as dev mode

3. **Detailed error messages and stack traces in API responses** (line 608): If DEV_MODE leaks to production, full stack traces are returned to all users, revealing internal paths, dependency versions, and database schema details.

**Recommendation:**
- Remove the environment-level `DEV_MODE` from production entirely. Use a build-time flag (`NODE_ENV=production`) that is compiled out, not a runtime environment variable that can be accidentally set.
- For account-level dev mode: never return verbose error details or stack traces to the client, even for dev-mode accounts. Instead, log a correlation ID and let devs look up the details in the logging backend. The debug panel should only show non-sensitive diagnostic info (request latency, cache hit/miss, etc.).
- Magic links and 2FA codes must never be logged anywhere, even in dev mode. Use the local SMTP trap (Mailpit) instead — that's already specified and is the correct approach.

## High Severity

### 4. Admin Console Lacks OAuth MFA Verification

**File:** requirements.md §8.9.2, line 688; §2.2, lines 52–54  
**Severity:** High

**Problem:** The requirements state: *"OAuth logins are considered inherently MFA-capable"* and *"OAuth-based logins (Microsoft, GitHub, Google, Apple) are considered sufficient for V1 and do not require an additional factor."*

This is incorrect. OAuth authentication does not guarantee the user has MFA enabled on their identity provider account. A GitHub account with only a password, or a Google account without 2-Step Verification, provides single-factor authentication. The admin console — which can suspend/delete accounts, modify feature flags, and view audit logs — is then protected by only a password.

The document acknowledges this as a future consideration (§12.1, line 1083: *"check `amr` claim to verify that OAuth providers have MFA enabled"*) but classifies it as deferred.

**Recommendation:**
- For V1: at minimum, require all admin accounts to also have email+password with 2FA enabled on Notebook.md, regardless of which provider they use to sign in. This ensures the admin always has a Notebook.md-controlled second factor.
- Better: check the `amr` (Authentication Methods References) claim in the OIDC token for Microsoft and Google. If MFA is not in the `amr` claim, deny admin access. GitHub does not provide `amr` but does expose `two_factor_authentication` on the user API endpoint — check that.

### 5. GitHub Webhook Lacks Replay Attack Protection

**File:** requirements.md §9.4, line 911  
**Severity:** High

**Problem:** The webhook endpoint description says *"verified via webhook secret signature"*, which addresses authenticity but the requirements do not mention timestamp validation or replay protection.

GitHub webhook signatures (HMAC-SHA256 of the payload) prove the payload came from GitHub, but without checking the `X-GitHub-Delivery` header for deduplication and the timestamp for freshness, an attacker who captures a valid webhook payload can replay it indefinitely.

A replayed `push` event could:
- Force unnecessary file tree refreshes, creating a denial-of-service on the backend
- In a more complex future webhook handler, trigger duplicate operations

**Recommendation:**
- Require timestamp checking: reject webhook deliveries older than 5 minutes (GitHub includes a timestamp in the payload).
- Track `X-GitHub-Delivery` IDs (UUID) in Redis with a short TTL (e.g., 10 minutes) and reject duplicates.
- Document these as explicit requirements for the webhook handler implementation.

### 6. Admin Self-Promotion and Missing Privilege Escalation Controls

**File:** requirements.md §8.9.3, lines 692–697; §9.6, line 927  
**Severity:** High

**Problem:** Multiple privilege escalation concerns:

1. **Any admin can create other admins** via `PATCH /admin/users/:id` (line 705/927: *"Update user flags (dev_mode, admin, suspended)"*). There's no distinction between admin tiers. A single compromised admin account can promote any number of additional admins.

2. **An admin can flag themselves as `is_dev_mode`**, enabling debug features on their own account that may expose additional data.

3. **No audit of admin actions separate from regular audit log**: Admin actions (promoting users, suspending accounts) go to the same audit log that admins can read (line 709: `GET /admin/audit-log`). An admin could potentially understand how to create actions that blend in, though they can't delete audit entries (audit logs are immutable per line 1005).

4. **Admin promotion via CLI** (line 693) requires container access (`docker exec`), which is good. But once one admin exists, they can create more via the API without any container access.

**Recommendation:**
- Restrict admin promotion: only the CLI (`promote-admin.js`) should be able to set `is_admin = true`. The `PATCH /admin/users/:id` endpoint should not be able to set the `is_admin` flag. This enforces that admin promotion always requires infrastructure-level access.
- If API-based admin creation is needed, require approval from multiple existing admins (quorum), or at minimum require re-authentication (password + 2FA) before privilege changes.
- Admin actions should be flagged distinctly in the audit log and should trigger real-time alerts (email/webhook to a security distribution list).

### 7. File Proxy Endpoints Vulnerable to Path Traversal

**File:** requirements.md §9.3, lines 898–904  
**Severity:** High

**Problem:** The API exposes file and folder endpoints with wildcard paths:
```
GET  /api/notebooks/:id/files/*path
PUT  /api/notebooks/:id/files/*path
POST /api/notebooks/:id/files/*path
DELETE /api/notebooks/:id/files/*path
```

These proxy requests to source system APIs. If the `*path` parameter is not strictly validated and canonicalized, path traversal attacks (e.g., `../../../etc/passwd` or `..%2F..%2F`) could:
- Access files outside the configured notebook root folder on OneDrive/Google Drive
- Access files in repos the user hasn't granted the GitHub App access to (if the backend constructs API calls using the path parameter)

The requirements do not mention path validation, canonicalization, or ensuring the resolved path stays within the notebook's configured root.

**Recommendation:**
- Add an explicit requirement: all `*path` parameters must be canonicalized (resolve `..`, `.`, double slashes) and validated to ensure the resulting path is within the notebook's configured root directory.
- Reject any path containing `..` segments after URL decoding.
- For GitHub, validate that the target repository matches the notebook's configured repository.

### 8. Broad OAuth Scopes Create Excessive Privilege

**File:** requirements.md §3.1, line 127; §2.5, line 93  
**Severity:** High

**Problem:** The requirements state tokens should be *"scoped to the minimum permissions required per provider"* (§2.5) but also specify:
- GitHub App permissions: *"Contents (read/write), Metadata (read), Pull Requests (read/write)"* (line 127)
- OneDrive: requires Microsoft Graph API access to OneDrive files
- Google Drive: requires Google Drive API access

For OneDrive and Google Drive, the requirements don't specify which scopes to request. The default/easy scopes for these providers (`Files.ReadWrite` for Microsoft Graph, `https://www.googleapis.com/auth/drive` for Google) grant access to the user's *entire* drive, not just the selected notebook folder.

Even for GitHub, the `Contents (read/write)` permission on the GitHub App grants write access to all installed repositories' contents, not just the notebook folder within a repo.

The requirements don't specify how to limit the blast radius — the app has write access to everything, but the user only selected one folder or repo as their notebook.

**Recommendation:**
- For OneDrive: use `Files.ReadWrite.AppFolder` if possible (limits to app-specific folder), or at minimum `Files.ReadWrite` with strict server-side path validation to ensure only the configured folder is accessed.
- For Google Drive: use the Drive API's `drive.file` scope, which limits access to files the app has created or the user has explicitly opened with the app. This is far safer than full drive access.
- For GitHub: the GitHub App's repo-level installation already provides some scoping (user selects which repos to install the app on). Document this as a requirement and validate on every API call that the target repo is one where the app is installed.
- Add an explicit security requirement: *"The backend must validate that every file operation targets only resources within the user's configured notebook boundaries, regardless of the OAuth scope granted."*

## Medium Severity

### 9. Cookie Consent Banner: Custom Implementation Risk

**File:** requirements.md §13.3, lines 1147–1152  
**Severity:** Medium

**Problem:** The requirements specify a *"custom cookie consent banner (built in-house, not a third-party library)"* with only two categories (Essential, Analytics) and "Accept" / "Manage Preferences" options.

GDPR and ePrivacy Directive compliance issues:
1. **No "Reject All" button**: GDPR guidance (and recent enforcement, e.g., CNIL fines against Google/Facebook) requires that rejecting cookies must be as easy as accepting them. A "Manage Preferences" button that requires additional clicks doesn't satisfy this. An explicit "Reject" button at the same level as "Accept" is increasingly required.
2. **Missing consent for OAuth provider cookies**: OAuth flows (Microsoft, Google, Apple) set their own cookies during the redirect. These may need disclosure.
3. **Consent storage**: The requirement says *"consent stored per user"* — but consent must be obtainable *before* the user is authenticated (they might not have an account yet). Consent should be stored in a cookie or localStorage, not tied to user identity.
4. **Building custom vs. using a vetted library**: Cookie consent law is nuanced and evolving. A custom implementation is more likely to have compliance gaps than a battle-tested library (e.g., CookieConsent by Orestbida, Osano, or similar open-source options). Given the stakes (GDPR fines up to 4% of revenue), this is an unnecessary risk for V1.

**Recommendation:**
- Add a "Reject All" button at the same level as "Accept All".
- Store consent in a first-party cookie (not tied to authentication) so it works for unauthenticated visitors.
- Strongly consider using an established open-source cookie consent library rather than building custom. If custom is preferred, have the implementation reviewed against GDPR/ePrivacy requirements by legal counsel.

### 10. No Rate Limiting Specified for File Proxy Endpoints

**File:** requirements.md §2.5 (line 97) vs §9.3  
**Severity:** Medium

**Problem:** Rate limiting is mentioned for auth endpoints (§2.5, line 97) but not for the file proxy endpoints (`/api/notebooks/:id/files/*path`). These endpoints proxy to external APIs (Microsoft Graph, Google Drive, GitHub) which have their own rate limits.

Without backend rate limiting:
- A compromised or malicious client could exhaust the user's API quota on the source system
- Aggressive auto-save or rapid operations could trigger rate limiting on the source system, causing cascading failures
- A malicious user could abuse the proxy to DOS the source system API from Notebook.md's IP, potentially getting the app's OAuth client ID banned

**Recommendation:**
- Add per-user rate limits on all file proxy endpoints.
- Implement backoff and retry logic that respects source system rate limit headers (`Retry-After`, `X-RateLimit-*`).
- Add circuit breakers per source system to prevent cascading failures.

### 11. Working Branch Name Contains Predictable User Info

**File:** requirements.md §4.10.3, line 311  
**Severity:** Medium

**Problem:** The working branch naming scheme is `notebook-md/<username>/<session-id>`. This exposes:
- That the user is using Notebook.md (may be undesirable for some users)
- The user's username
- A session identifier

On public GitHub repos, these branches are visible to anyone. If `session-id` is predictable or sequential, it could leak session volume information. If the branch isn't cleaned up (even though "delete branch on publish" defaults to on), it persists.

**Recommendation:**
- Use a less information-dense branch name, e.g., `notebook-md/<random-id>`.
- Ensure session IDs are cryptographically random UUIDs, not sequential.
- Consider whether the branch name format should be configurable or at least obscured for public repos.

### 12. TOTP Secret and Recovery Codes Encryption Key Management Unspecified

**File:** requirements.md §10.1, lines 949–952  
**Severity:** Medium

**Problem:** The data model stores `totp_secret_enc` and `recovery_codes_enc` as encrypted BYTEA fields. OAuth tokens are described as using envelope encryption with AES-256 and a KMS-managed key (§2.5, line 92). However, the TOTP secrets and recovery codes are not explicitly included in that envelope encryption scheme.

If TOTP secrets are encrypted with a key that's stored alongside the database (e.g., in an environment variable rather than a KMS), a database compromise also yields the encryption key, making the encryption worthless.

Additionally, `recovery_codes_enc` is described as *"encrypted JSON array of hashed recovery codes"*. If the recovery codes are hashed (which is correct — they should be bcrypt-hashed like passwords), they don't need encryption on top. If they're encrypted but not hashed, then a key compromise reveals all recovery codes. The requirement is ambiguous.

**Recommendation:**
- Explicitly require that TOTP secrets use the same KMS-based envelope encryption as OAuth tokens.
- Recovery codes should be hashed (bcrypt), not encrypted. A hash is one-way and protects the codes even if the database is fully compromised. Update the column description to `recovery_codes_hash` and remove the `_enc` suffix to clarify intent.

### 13. No Content Security Policy Details

**File:** requirements.md §11.2, line 1041  
**Severity:** Medium

**Problem:** The requirements mention *"Content Security Policy (CSP) headers"* as a single line item without any specifics. Given the app:
- Renders user-provided Markdown as HTML (XSS vector)
- Renders LaTeX math via KaTeX (requires `unsafe-eval` or specific CSP exceptions in some implementations)
- May embed external images and media referenced by URL in Markdown documents
- Uses OAuth redirect flows to external providers

A poorly configured CSP could either:
- Be too permissive and fail to prevent XSS (e.g., `unsafe-inline`, `unsafe-eval`, `*` sources)
- Be too restrictive and break core features (math rendering, external images, OAuth redirects)

**Recommendation:**
- Add a specific CSP requirement: define a strict base policy with nonce-based script loading, and explicitly document which exceptions are needed and why (e.g., KaTeX rendering, external image sources).
- External images referenced in Markdown should be proxied through the backend or loaded via `img-src` with an explicit allowlist, not `img-src *`.
- Math rendering library choice (KaTeX vs MathJax) should consider CSP compatibility — KaTeX is generally more CSP-friendly than MathJax.

### 14. Markdown Rendering XSS Surface

**File:** requirements.md §4.1–4.3, §11.2 line 1045  
**Severity:** Medium

**Problem:** The requirements mention *"Input sanitization for all user-provided data (XSS prevention in rendered Markdown)"* (line 1045) but don't specify a sanitization strategy.

The WYSIWYG editor renders Markdown as live HTML. Markdown can contain:
- Raw HTML (GFM allows inline HTML)
- JavaScript event handlers in HTML attributes (`<img onerror="...">`)
- SVG with embedded scripts (`<svg onload="...">`)
- Link targets with `javascript:` URIs (`[click](javascript:alert(1))`)
- Data URIs that could contain HTML/JS

Since the app displays user content from external sources (OneDrive, Google Drive, GitHub), a malicious file could be crafted to exploit the renderer. This is especially dangerous given Finding #2 (OAuth tokens in the browser) — XSS + token theft = full compromise.

**Recommendation:**
- Add an explicit requirement to use a proven HTML sanitization library (e.g., DOMPurify) on all rendered Markdown output.
- Disable raw HTML in the Markdown parser, or sanitize it aggressively (strip all event handlers, `javascript:` URIs, `data:` URIs for non-image content, `<script>`, `<style>`, `<iframe>`, `<object>`, `<embed>`, `<form>`, `<input>`, SVG scripts).
- Tiptap/ProseMirror provides some protection via its schema (only known node types are rendered), but this must be explicitly verified and not assumed.

## Low Severity

### 15. No Specification for CORS Configuration

**File:** requirements.md §8.2  
**Severity:** Low

**Problem:** The architecture mentions the backend acts as a proxy "for sources where CORS prevents direct browser access" but doesn't specify the CORS policy for the Notebook.md API itself.

Without a strict CORS policy, the API at `api.notebookmd.io` could be called from any origin, enabling cross-origin attacks against authenticated users.

**Recommendation:**
- Add an explicit requirement: the API must set `Access-Control-Allow-Origin` to only the exact origins of the web app (`notebookmd.io`) and admin console (`admin.notebookmd.io`). Never use `*` or reflect the `Origin` header.
- `Access-Control-Allow-Credentials: true` must only be set with explicit origin matching.

### 16. Container Image Tag Hygiene

**File:** requirements.md §8.14.3, line 854  
**Severity:** Low

**Problem:** The requirements state images are tagged with git SHA and semantic version, and *"`latest` tag points to current production."* Using `latest` in production is an anti-pattern — it's mutable and makes it impossible to guarantee which image is actually running. If a container restarts and pulls `latest`, it could get a different image than what was deployed.

**Recommendation:**
- Always deploy with immutable tags (git SHA or semantic version). Never reference `latest` in production deployment configurations.
- `latest` can exist in the registry as a convenience for development, but production Container Apps revisions should always pin to a specific version tag.

### 17. No Session Binding or Device Fingerprinting

**File:** requirements.md §2.6, lines 100–108  
**Severity:** Low

**Problem:** The session management section describes refresh token rotation and family-based revocation, which is good. However, sessions are not bound to any device or client characteristics.

A stolen refresh token can be used from any device, any IP, any location until it expires or the rotation detects theft. Since "Remember Me" extends sessions to 30 days, a stolen token has a long window of exploitation before detection.

**Recommendation:**
- Consider binding sessions to client fingerprint (IP range, User-Agent) and requiring re-authentication if these change significantly.
- For high-sensitivity actions (linking/unlinking providers, accessing admin console), require a step-up authentication regardless of session validity.

---

### 18. Audit Log Lacks Integrity Protection

**File:** requirements.md §10.1, lines 997–1005  
**Severity:** Low

**Problem:** The audit log is described as immutable (no `updated_at` column), but there's no protection against a compromised admin or database-level access deleting or modifying records. PostgreSQL doesn't enforce append-only tables — a database superuser or anyone with `DELETE` privileges can remove rows.

**Recommendation:**
- Use a separate database user for audit log writes that has only `INSERT` privileges (no `UPDATE`, `DELETE`).
- Consider hash-chaining audit log entries (each entry includes a hash of the previous entry) to make tampering detectable.
- For high-security environments, consider streaming audit logs to an external, immutable log store (e.g., Azure Immutable Blob Storage) in addition to PostgreSQL.
